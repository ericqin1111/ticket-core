## 问题与机遇

`EPIC-TKT-002 / Stage B` 的任务不是把履约失败处理做成复杂编排，而是在 `feat-TKT002-01` 已经建立单次履约执行骨架之后，为常见失败补齐最小自动恢复能力。

当前已冻结的上游事实是：

* `Fulfillment` 已具备 `PENDING -> PROCESSING -> SUCCEEDED / MANUAL_PENDING` 的执行与治理基线。
* `PROCESSING` 是持久治理态，不是瞬时态。
* Stage A 已不再把单次执行失败直接写成 `FAILED`，而是先收敛到治理落点。
* 若外部调用结果未知，系统当前治理底线是“先避免盲重试”，不允许为了追求自动恢复而制造重复履约风险。

因此，本 feat 的机会在于：把“能跑一次”的履约系统推进到“面对常见抖动仍可有限自动收敛”的阶段，同时明确哪些失败可以自动恢复、哪些失败必须终止并转人工。

## 需求

### 业务愿景概述

让系统在履约失败后，能够基于稳定的失败分类规则，把可安全恢复的对象推进到 `RETRY_PENDING` 并执行有限自动重试，把不可自动恢复或结果不确定的对象推进到 `MANUAL_PENDING`，从而把自动恢复、人工接管和最终失败终局清晰拆开。

### 领域概念辞典

| 概念 | 定义 |
|:--|:--|
| `Fulfillment Attempt` | 某条履约的一次具体执行尝试，是 Stage B 讨论重试次数与结果分类的基本单位。 |
| `Fulfillment.RETRY_PENDING` | 当前对象允许继续自动恢复，等待下一次自动重试调度。 |
| `Fulfillment.MANUAL_PENDING` | 当前对象已退出自动恢复路径，等待人工接管或人工裁决。 |
| `Fulfillment.FAILED` | 经人工治理后确认终止的最终失败态，不再表示“某次自动执行失败”。 |
| `Retryable Technical Failure` | 明确判断为短时环境抖动、稍后重试有现实成功概率，且重复执行风险可控的失败。 |
| `Final Business Rejected` | 已有明确业务依据说明当前履约应直接终止，且不存在人工补救或人工再裁决价值的拒绝。 |
| `Manual Review Required` | 自动流程已无法继续，但业务上仍存在人工补救、人工确认或人工裁决价值的拒绝。 |
| `Uncertain Result` | 已发起外部执行，但系统无法确认外部是否已成功；若盲重试可能造成重复履约。 |
| `Fast Retry` | 首次失败后，为短时抖动提供的一次快速自动重试，不代表可无限连续重试。 |
| `Backoff Retry` | 在快速重试之外，为可重试失败提供的有限次延迟重试。 |
| `Retry Budget` | 单条履约在自动恢复阶段允许消耗的总自动重试额度。 |
| `Failure Reason Code` | 用于运营识别和规则判断的稳定失败分类码，不等同于 provider 原始错误码。 |
| `Manual Recovery Entry` | 当对象被判定为不可自动恢复时，进入 Stage C 人工操作前的稳定治理落点。 |

### 主干流程

1. 某次 `Fulfillment Attempt` 执行失败后，系统必须先判断失败属于哪一类，而不是对所有失败统一重试。
2. 若失败被判定为 `Retryable Technical Failure`，系统将对象推进到 `RETRY_PENDING`，允许在人工介入前执行有限自动重试。
3. 对可重试失败，首次失败后允许一次快速重试，以覆盖短时网络抖动、网关瞬时异常、临时限流等常见场景。
4. 若快速重试仍失败，系统可继续按有限次退避策略执行后续自动重试；在重试预算耗尽前，对象持续处于 `RETRY_PENDING` 治理语义中。
5. 若失败被判定为 `Final Business Rejected`，系统不得自动重试，可直接收敛到 `FAILED`。
6. 若失败被判定为 `Manual Review Required`，系统不得自动重试，应收敛到 `MANUAL_PENDING`，等待人工接管或人工确认终止。
7. 若失败被判定为 `Uncertain Result`，系统不得盲目自动重试，而应优先进入 `MANUAL_PENDING` 或后续专门核查路径。
8. 若自动重试预算耗尽，系统必须退出自动恢复路径并收敛到 `MANUAL_PENDING`，而不是继续后台循环或直接混写成最终 `FAILED`。
9. 每次失败分类、每次自动重试决策、每次进入 `RETRY_PENDING` / `MANUAL_PENDING` 或直接进入 `FAILED` 的治理动作，都必须留下可审计、可诊断的结构化记录。

### 异常与边界规则

* Stage B 允许“有限自动恢复”，但不允许把所有失败都默认视为可重试。
* `Final Business Rejected` 与 `Manual Review Required` 一旦成立，都不应因为调度重复或系统重启而重新进入自动重试。
* `Retryable Technical Failure` 只适用于“重试有意义且重复执行风险可控”的对象。
* `Uncertain Result` 必须单独处理，不能简单归入环境失败；否则容易造成重复出票、重复发码或重复交付。
* `PROCESSING` 超时对象必须被识别并收敛，不能无限停留在处理中。
* 对 `PROCESSING` 僵死对象，平台不要求一定能精确识别“未开始 / 进行中 / 已完成”真实状态；真正的治理标准是“是否有充分证据支持安全自动下一步”。
* `PROCESSING` 僵死后的默认出口是 `MANUAL_PENDING`；只有在能确认“尚未成功且可安全重试”时，才允许转入 `RETRY_PENDING`。
* 快速重试是受限能力，不代表连续多次即时重试；避免把局部抖动放大为热点自旋。
* 自动重试总预算必须很小，目标是覆盖常见抖动，而不是长期占用系统等待偶然恢复。
* 若自动重试已耗尽，系统必须稳定退出自动恢复路径并进入 `MANUAL_PENDING`，不得在后台无限循环。
* `FAILED` 不再作为自动流程失败落点，而只保留给 Stage C 人工治理后的最终失败结论。
* 幂等仍是本阶段底线：重复调度、重复提交、超时回放都不得造成双成功或双终局。

## 当前阶段明确不处理

* Stage C 的人工操作集合、权限边界与操作审计细则。
* provider 结果核查接口、补单接口或自动补记结果的复杂恢复编排。
* 多 provider 路由下按供应商能力差异化重试。
* 取消、退款、逆向履约或部分履约场景。
* 面向外部渠道暴露更细粒度的失败可视化 contract。

## 已确认决议补充说明

### D1. 为什么必须把 `Uncertain Result` 单独分类

已确认选择 `B: 至少三类`。

原因如下：

* 票务履约不是普通弱副作用调用，一旦外部系统已经成功出票、锁码或完成资源分配，再次执行可能造成重复交付。
* “网络超时”或“回包缺失”并不天然等于“外部未成功”；如果平台没有单独的结果未知语义，下游实现很容易把这类场景直接归入环境失败并发起自动重试。
* 因此 Stage B 必须把“确定失败”和“结果未知”拆开，让治理语义先收敛，再谈自动恢复动作。

对后续设计的约束是：

* 失败分类至少包含 `Final Business Rejected / Manual Review Required`、`Retryable Technical Failure`、`Uncertain Result` 三类治理语义。
* `Retryable Technical Failure` 应进入 `RETRY_PENDING`；`Uncertain Result` 不得直接进入自动重试队列。
* provider 原始错误可以保留，但不能越过平台一级分类直接驱动自动决策。

### D3. 为什么只允许 1 次快速重试

已确认选择 `B: 支持 1 次快速重试`。

原因如下：

* 短时网络抖动、瞬时 5xx、短暂限流在票务履约里确实存在，一次快速重试通常有业务价值。
* 但如果放开 2-3 次连续快速重试，系统很容易在同一短时间窗口内对同一对象反复冲击 provider，放大热点、吞掉诊断信号，并加剧幂等压力。
* Stage B 的目标是“最小恢复”，不是把调度系统变成复杂重试引擎，因此快速重试应限制在 1 次。

对后续设计的约束是：

* 快速重试只适用于 `Retryable Technical Failure`。
* 快速重试失败后，不得继续做连续秒级重试。
* 后续若仍允许自动恢复，应继续停留在 `RETRY_PENDING` 并进入有限次退避重试；若不再允许自动恢复，应退出到 `MANUAL_PENDING`。

### D5. 为什么 `Uncertain Result` 必须转人工治理

已确认选择 `B: 禁止盲重试，转人工治理`。

原因如下：

* 在结果未知场景下，系统缺少“外部尚未成功”的稳定依据，此时再执行一次履约动作，本质上是在拿重复履约风险换取自动成功率。
* 对票务履约而言，重复出票、重复发码、重复占用资源的代价通常高于“需要人工核查一次”。
* 所以 Stage B 的治理目标不是把所有异常都自动跑通，而是先保证系统不会越治理越错。

对后续设计的约束是：

* 结果未知对象必须有清晰的人工治理落点，即 `MANUAL_PENDING`。
* 自动调度与人工治理之间的状态边界应在 Stage C 继续收敛。
* 若后续确需引入自动核查或自动补记能力，应作为单独能力设计，而不是在 Stage B 默认承诺。

### D4. 为什么业务拒绝不能一刀切

已确认选择 `C: 按是否仍有人工处理价值拆分`。

原因如下：

* 有些业务拒绝已经足够明确，例如履约窗口已过、订单关键条件永久无效、供应商明确不可履约且业务上不存在替代路径；这类对象继续停在 `MANUAL_PENDING` 只会增加人工噪音，更适合直接进入 `FAILED`。
* 另一些业务拒绝虽然自动流程不能继续，但仍可能存在人工补救空间，例如修正上游映射、补齐资料、切换人工票源或人工确认终止；这类对象若直接写成 `FAILED`，会过早抹掉人工收尾空间。
* 因此 Stage B 需要把“自动不能继续”与“整条业务已最终终止”区分开，不能把所有业务拒绝都压成同一个出口。

对后续设计的约束是：

* `Final Business Rejected` 可直接进入 `FAILED`。
* `Manual Review Required` 必须进入 `MANUAL_PENDING`。
* 是否存在人工处理价值，应由稳定业务规则判断，而不是由实现层临时拍脑袋决定。

### 为什么 Phase 1 接受多数失败进入人工治理

这是当前阶段有意接受的取舍，不视为设计偏差。

原因如下：

* 票务履约的重复执行风险高，误重试代价通常大于“先转人工确认一次”的代价。
* 平台对外部真实执行状态的可观测性有限，很多失败并没有足够证据支持安全自动恢复。
* 当前 Phase 1 目标是先把正确性、可收敛性和治理边界建稳，而不是提前承诺高自动恢复率。

这项取舍的边界也必须明确：

* 可以接受“大部分失败进入 `MANUAL_PENDING`”。
* 不接受“因为规则不清，所有不确定问题都粗暴甩给人工”。
* 因此进入人工治理的对象，必须有稳定的分类依据、明确的进入条件和清晰的后续接管语义。

### D8. 为什么 stuck processing 不要求“三态精确识别”

已确认选择 `B: 以是否可安全自动下一步为治理标准`。

原因如下：

* “业务未开始 / 业务进行中 / 业务已完成”更像是业务真实状态，而不是平台在任意 provider、任意异常下都能稳定观测到的事实。
* 如果 Stage B 假设所有超时对象都能被准确分类，文档会隐含平台具有并不存在的可观测性，后续实现很容易失真。
* 对票务履约来说，真正重要的不是把每个 stuck 对象说得很漂亮，而是避免因为误判而触发盲重试。

对后续设计的约束是：

* `PROCESSING` 超时对象必须有明确治理出口，不能无限挂起。
* 若能确认“未成功且可安全重试”，允许转 `RETRY_PENDING`。
* 若无法确认真实进度，或无法证明重试安全，默认转 `MANUAL_PENDING`。
* 本阶段不承诺 provider 级自动核查、自动补记或更细颗粒度 stuck 分类算法。

### 为什么 Stage B 要采用双治理态，而不是继续复用 `FAILED`

本轮回改确认采用“双治理态 + 最终失败态”的表达方式，即：

* `RETRY_PENDING`：仍允许系统自动恢复。
* `MANUAL_PENDING`：自动流程已停止，等待人工接管。
* `FAILED`：人工治理后确认终止的最终失败。

变更理由如下：

* 如果继续用 `FAILED` 同时表达“本次 attempt 失败”“自动恢复耗尽”“待人工处理”“最终确认失败”，业务语义会持续打架。
* Stage B 的核心就是把“系统还要不要继续自动处理”说清楚；因此需要一个稳定的自动恢复治理态 `RETRY_PENDING`。
* Stage C 的核心是把人工介入入口与人工终局说清楚；因此需要一个稳定的人工接管治理态 `MANUAL_PENDING`，并把 `FAILED` 留给人工确认后的最终结论。

这套拆分虽然比早期简化模型多一个状态，但换来的是：

* 自动恢复和人工接管边界更清晰。
* `Attempt` 层失败与 `Fulfillment` 层终局不再混淆。
* 后续对外 contract、审计和运营口径更容易保持一致。
