# Headless Ticketing Fulfillment System

## 1. 项目定义

本项目是一个**以内核为核心、带最小完整业务外壳的 Headless Ticketing Fulfillment System**。

它不是面向终端消费者的购票网站，也不承担内容分发、营销增长或会员运营职责。它面向主办方、自营渠道、第三方分销渠道与内部运营团队，提供一套可被真实业务消费的票务交易与履约底座。

系统的中心不是页面，而是**交易履约内核**。项目的核心价值在于：在高并发、重复请求、回调乱序、链路超时、局部故障等非理想条件下，依然能够维持库存安全、状态一致与履约可恢复。

一句话概括：

> Not a ticketing website, but the fulfillment backbone behind ticket sales.

## 2. 要解决的核心问题

本项目聚焦三个真实且高价值的问题：

### 2.1 高并发下的库存安全

在热门活动、秒杀、集中放票等场景下，系统必须控制库存竞争，避免超卖、重复占票与异常释放。

### 2.2 多渠道协同下的交易一致性

不同销售渠道会共享同一套活动资产与履约能力。渠道之间可以隔离接入、独立调用，但底层库存、订单状态与履约结果必须统一。

### 2.3 异常条件下的履约可恢复

支付超时、回调重放、消息重复、服务抖动、人工干预等情况都必须被视为常态。系统需要通过幂等、超时控制、补偿修复与审计追踪，保证状态最终落回可解释、可追责、可继续处理的轨道。

## 3. 系统边界

### 3.1 系统负责的范围

- 活动、场次、票种、销售规则、价格规则的基础配置
- 库存查询、锁票、占座、下单、支付确认、出票通知
- 订单生命周期管理与履约状态流转
- 超时释放、取消、退款、人工纠偏等逆向流程
- 渠道接入、Webhook 通知、调用审计
- 基础风控、运维治理、对账基线与可观测性支撑

### 3.2 系统明确不负责的范围

- C 端导购首页、内容运营、推荐分发
- 会员成长体系、CRM、用户画像平台
- 复杂营销中台与活动编排平台
- 通用型 BI / 数据分析平台
- 脱离票务领域的泛化工作流引擎

边界原则很明确：**以交易履约为中心，只保留让内核能够被真实业务消费所必需的最小外壳。**

## 4. 核心业务主链路

项目围绕以下主链路构建：

1. 渠道查询活动、场次、票种与可售库存
2. 渠道发起锁票或占座请求
3. 交易内核完成库存预占用并创建待支付订单
4. 支付结果驱动订单进入确认态
5. 系统完成出票或确票，并向渠道异步通知
6. 未支付超时订单自动释放库存
7. 异常订单通过补偿机制回归可处理状态
8. 退款、取消、人工干预进入逆向履约链路
9. 全流程沉淀审计记录，支撑追责、对账与运营治理

README、Epic、RFC 与后续模块设计都应围绕这条主链路展开。任何新能力如果无法挂接到这条链路上，就需要重新评估其必要性。

## 5. 系统组织方式

本项目采用**内核主导型系统（Kernel-led System）**组织方式，整体分为三层。

### 5.1 第一层：交易履约内核

这是项目的主角，也是最核心的竞争力所在。

职责包括：

- 库存控制与资源预占用
- 锁票、释放、选座一致性校验
- 订单状态机与状态迁移约束
- 幂等控制与重复请求拦截
- 超时回收与补偿修复
- 履约事件记录与审计追踪

这一层回答的问题是：**票会不会卖错，状态会不会乱，异常后能不能恢复。**

### 5.2 第二层：最小业务外壳

这层不是为了做“大平台”，而是为了让内核真正能够被使用。

职责包括：

- 活动、场次、票种、价格与销售规则配置
- 渠道接入 API 与鉴权契约
- 订单查询、库存视图与基础运营查询
- Webhook / 异步通知
- 基础退款、取消与人工干预接口

这一层回答的问题是：**外部系统如何接入，业务如何完成一次完整交易。**

### 5.3 第三层：治理与可靠性支撑

这层用于保障系统可以长期运行，但不应反客为主。

职责包括：

- 限流、降级、隔离与熔断
- 基础风控与恶意流量治理
- 对账、审计与运维操作留痕
- 脱敏、加密与合规治理
- 监控、告警、指标与追踪
- 数据归档与生命周期管理

这一层回答的问题是：**系统能不能稳定跑下去，并在出问题时被管理与追溯。**

## 6. 核心模块

在当前项目边界下，系统至少应包含以下六个核心模块：

### 6.1 Catalog

负责活动、场次、票种、价格规则、销售时间窗与基础售卖配置。

### 6.2 Inventory

负责库存池、可售视图、预占用、释放、配额控制与座位资源管理。

### 6.3 Order

负责订单创建、状态机、幂等控制、支付确认、超时回收与补偿入口。

### 6.4 Fulfillment

负责出票、确票、通知、核销以及退款 / 取消等逆向履约处理。

### 6.5 Channel Gateway

负责渠道身份认证、OpenAPI 契约、Webhook 投递、调用限流与接入审计。

### 6.6 Operations & Governance

负责人工纠偏、审计日志、基础风控、对账基线、观测与运维治理能力。

这六个模块构成项目的最小完整集。少于这个集合，项目会退化为“只有内核的技术 demo”；大幅超出这个集合，则容易过早膨胀成边界失控的平台化系统。

## 7. 设计原则

后续开发应始终遵循以下原则。

### 7.1 内核优先

所有设计决策首先服务于库存安全、状态一致与履约可恢复，而不是优先追求功能面广度。

### 7.2 主链路优先

优先完成查询、锁票、下单、支付确认、出票通知、超时释放、退款取消这条主链路，再扩展外围能力。

### 7.3 API First / Headless First

系统能力通过稳定契约向外暴露，不与单一前台形态绑定。

### 7.4 面向异常设计

默认假设重复请求、乱序回调、消息重放、网络抖动、部分依赖失败都会发生。异常路径不是补丁，而是主设计对象。

### 7.5 最小完整，而非大而全

外围能力必须服务于交易履约闭环，避免过早引入与主价值无关的大型平台化建设。

### 7.6 可审计、可追责、可恢复

关键状态变化必须可记录、可解释、可追溯。任何自动化补偿都不能以牺牲可观测性为代价。

## 8. 可靠性目标表达方式

本项目追求强可靠，但后续文档表达应避免脱离边界的绝对化宣传。

推荐使用如下表达口径：

- 核心库存账本与订单状态迁移遵循严格状态约束
- 在定义的并发与故障模型内保证不超卖
- 通过幂等、超时控制与补偿任务实现异常状态可恢复
- 通过审计留痕与对账基线保证关键流程可追责
- 通过分级脱敏、加密落盘与权限控制降低 PII 暴露风险

不推荐直接使用没有边界说明的措辞，例如“绝对一致”“绝对安全”“100% 无条件正确”。

## 9. 后续开发优先级

项目后续开发应按以下顺序推进：

1. 先固化主链路与领域边界
2. 再定义核心状态机与库存一致性模型
3. 再拆分模块职责、接口契约与异步事件流
4. 最后逐步补齐治理、风控、对账与合规能力

换句话说，先把它做成一个**能稳定完成真实票务履约闭环的系统**，再把它逐步做成一个**治理完善的票务基础设施**。

## 10. 项目定位总结

本项目最终希望形成的，不是一个炫技的分布式 demo，也不是一个无边界膨胀的平台集合，而是：

> 一个以交易履约内核为核心、为多渠道售票场景提供库存安全、状态一致性与异常可恢复能力的 Headless Ticketing Fulfillment System。

这份 README 不是宣传页，而是后续架构设计、需求拆解、RFC 编写与实现取舍的基线说明。
